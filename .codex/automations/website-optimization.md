# Astro 个人网站每日优化任务

本任务由 `Codex网站优化自动化.md` 整理而来，用作 Codex Automations / 项目自动化的可复制任务定义。默认每天 `23:30 UTC` 在当前项目目录运行，也允许手动触发。

## 任务身份

你是这个 Astro 个人网站的长期网页优化代理。

## 项目目标

根据当前网站已有的后朋克、雨夜、冷色调、lofi、赛博末世、屋顶城市风格，持续优化网站的样式、展示形式、交互、图片、贴图、背景层次、动态视觉和内容表达。

## 固定风格约束

- 后朋克
- 阴郁写实
- 雨夜城市
- 屋顶、loft、远处城市
- 冷色调、蓝黑、雾灰、暗紫
- 少量霓虹蓝、霓虹粉、暖黄窗光
- lofi、潮湿、孤独、安静
- 不要企业官网模板感
- 不要后台管理系统感
- 不要高饱和电竞风
- 不要纯动漫插画风
- 动效要克制，优先轻量性能

## 每次运行流程

### 1. 安全检查

- 查看 `git status`。
- 如果运行前已经存在未提交修改，不要直接覆盖，也不要提交这些修改。
- 先生成报告，说明当前工作区不干净，需要用户确认后再继续。

### 2. 检查当前网站代码

- 读取 `src/pages/index.astro`。
- 读取 `src/components/CyberpunkHero.astro`。
- 读取 `src/layouts/BaseLayout.astro`。
- 读取 `src/styles/global.css`。
- 检查 `public/assets` 下的图片和贴图。
- 找出代码冗余、性能风险、移动端问题、文字可读性问题、图片加载问题、样式不统一问题。

### 3. 生成本次优化建议

- 先列出 3 到 6 条建议。
- 每条建议说明为什么要改、预计影响、风险。
- 优先选择风险低、效果明显、符合现有风格的优化。

### 4. 执行本次优化

- 每次只做一组主题明确的优化。
- 可以新增图片、贴图、背景样式、动态图标或交互效果。
- 如果需要生成图片，图片必须复制到 `public/assets` 下，并压缩为 WebP。
- 不要引用本机临时目录里的图片。
- 不要引入 React、Vue、Svelte、Tailwind 或大型依赖。
- 优先使用 Astro、HTML、CSS、少量原生 JavaScript。
- 动效必须考虑性能，不要大量使用 `fixed`、`blur`、`mix-blend-mode`、持续 `requestAnimationFrame`。

### 5. 本地验证

- 运行 `npm.cmd run build`；在 Linux/macOS 或 CI 中可使用 `npm run build`。
- 如果构建失败，修复后重新构建。
- 启动本地预览或使用现有截图脚本进行截图。
- 截图文件名必须包含日期时间和本次版本号，例如：
  - `screenshots/auto-2026-06-05-2330-v001-home.png`
  - `screenshots/auto-2026-06-05-2330-v001-works.png`

### 6. 截图判断

- 检查首页 Hero。
- 检查至少 2 个后续分区。
- 判断是否存在文字重叠、视觉拥挤、图片加载失败、横向滚动、移动端严重问题、动效过重。
- 如果截图显示明显问题，继续修复并重新截图。
- 循环直到本轮优化达到可交付状态。

### 7. 查询资料并填充内容

- 根据网站当前主题查询相关资料。
- 可参考后朋克、lo-fi、工业音乐、赛博朋克视觉、雨夜城市、个人网站展示方式等方向。
- 文案必须和网站风格一致，克制、冷静、阴郁，不要营销腔。
- 填充内容时优先保证可读性和页面节奏。

### 8. 生成总结报告

- 在 `reports/automation` 下生成 Markdown 报告。
- 文件名包含日期时间和版本号，例如：`reports/automation/2026-06-05-2330-v001.md`。
- 报告必须包含：
  - 本次运行时间
  - 本次版本号
  - 检查到的问题
  - 采纳的优化建议
  - 实际修改文件
  - 新增资源
  - 构建结果
  - 截图文件列表
  - 视觉判断结论
  - 性能风险
  - 下一次建议优化内容
  - 是否已提交和推送

### 9. 自动提交并推送

- 只提交本次自动化生成或修改的文件。
- 默认提交源码、资源和 `reports/automation` 下的报告。
- `screenshots/` 当前被 `.gitignore` 忽略，截图默认本地留存，报告中记录截图路径。
- 提交信息格式：`Auto optimize website vNNN`。
- 推送到当前分支。
- 如果 `git push` 失败，在报告中说明失败原因，不要反复强推。

## 重要限制

- 不要删除用户未确认的文件。
- 不要重置仓库。
- 不要提交运行前已经存在的未提交修改。
- 不要把 `node_modules`、`dist`、临时缓存或本机绝对路径资源提交进去。
- 不要提交 `hatch-pet-runs`、`node_modules/.vite` 等临时生成目录。
- 不要为了视觉效果牺牲明显性能。

- 第一次自动化运行：`v001`
- 第二次自动化运行：`v002`
