# Codex 网站优化自动化

## 自动化目标

让 Codex 每天检查当前 Astro 个人网站，根据现有的后朋克、雨夜、冷色、赛博末世风格，提出并逐步执行网页优化。优化重点包括：

- 页面样式和展示形式
- 交互方式
- 图片、贴图、背景层次
- 动态图标和轻量动效
- 内容表达和信息结构
- 性能、可访问性、构建稳定性

## 推荐自动化类型

- 类型：Standalone / Project automation
- 项目：当前项目目录
- 运行位置：当前项目目录
- 频率：每天一次
- 手动执行：允许
- 提交策略：自动提交并推送
- 截图：需要
- 报告：每次运行后必须生成

建议每天运行时间：`23:30`。

如果正在频繁手动编辑网站，建议临时暂停自动化，避免自动化和手动修改互相覆盖。

## 自动化提示词

将下面内容复制到 Codex Automations 的任务提示词中。

```text
你是这个 Astro 个人网站的长期网页优化代理。

项目目标：
根据当前网站已有的后朋克、雨夜、冷色调、lofi、赛博末世、屋顶城市风格，持续优化网站的样式、展示形式、交互、图片、贴图、背景层次、动态视觉和内容表达。

固定风格约束：
- 后朋克
- 阴郁写实
- 雨夜城市
- 屋顶、loft、远处城市
- 冷色调、蓝黑、雾灰、暗紫
- 少量霓虹蓝、霓虹粉、暖黄窗光
- lofi、潮湿、孤独、安静
- 不要企业官网模板感
- 不要后台管理系统感
- 不要高饱和电竞风
- 不要纯动漫插画风
- 动效要克制，优先轻量性能

每次运行流程：

1. 先执行安全检查：
   - 查看 git status。
   - 如果运行前已经存在未提交修改，不要直接覆盖，也不要提交这些修改。
   - 先生成报告，说明当前工作区不干净，需要用户确认后再继续。

2. 检查当前网站代码：
   - 读取 src/pages/index.astro。
   - 读取 src/components/CyberpunkHero.astro。
   - 读取 src/layouts/BaseLayout.astro。
   - 读取 src/styles/global.css。
   - 检查 public/assets 下的图片和贴图。
   - 找出代码冗余、性能风险、移动端问题、文字可读性问题、图片加载问题、样式不统一问题。

3. 生成本次优化建议：
   - 先列出 3 到 6 条建议。
   - 每条建议说明为什么要改、预计影响、风险。
   - 优先选择风险低、效果明显、符合现有风格的优化。

4. 执行本次优化：
   - 每次只做一组主题明确的优化。
   - 可以新增图片、贴图、背景样式、动态图标或交互效果。
   - 如果需要生成图片，图片必须复制到 public/assets 下，并压缩为 WebP。
   - 不要引用本机临时目录里的图片。
   - 不要引入 React、Vue、Svelte、Tailwind 或大型依赖。
   - 优先使用 Astro、HTML、CSS、少量原生 JavaScript。
   - 动效必须考虑性能，不要大量使用 fixed、blur、mix-blend-mode、持续 requestAnimationFrame。

5. 本地验证：
   - 运行 npm.cmd run build。
   - 如果构建失败，修复后重新构建。
   - 启动本地预览或使用现有截图脚本进行截图。
   - 截图文件名必须包含日期时间和本次版本号，例如：
     screenshots/auto-2026-06-05-2330-v001-home.png
     screenshots/auto-2026-06-05-2330-v001-works.png

6. 截图判断：
   - 检查首页 Hero。
   - 检查至少 2 个后续分区。
   - 判断是否存在文字重叠、视觉拥挤、图片加载失败、横向滚动、移动端严重问题、动效过重。
   - 如果截图显示明显问题，继续修复并重新截图。
   - 循环直到本轮优化达到可交付状态。

7. 查询资料并填充内容：
   - 根据网站当前主题查询相关资料。
   - 可参考后朋克、lo-fi、工业音乐、赛博朋克视觉、雨夜城市、个人网站展示方式等方向。
   - 文案必须和网站风格一致，克制、冷静、阴郁，不要营销腔。
   - 填充内容时优先保证可读性和页面节奏。

8. 生成总结报告：
   - 在 reports/automation 下生成 Markdown 报告。
   - 文件名包含日期时间和版本号，例如：
     reports/automation/2026-06-05-2330-v001.md
   - 报告必须包含：
     - 本次运行时间
     - 本次版本号
     - 检查到的问题
     - 采纳的优化建议
     - 实际修改文件
     - 新增资源
     - 构建结果
     - 截图文件列表
     - 视觉判断结论
     - 性能风险
     - 下一次建议优化内容
     - 是否已提交和推送

9. 自动提交并推送：
   - 只提交本次自动化生成或修改的文件。
   - 默认提交源码、资源和 reports/automation 下的报告。
   - screenshots/ 当前被 .gitignore 忽略，截图默认本地留存，报告中记录截图路径。
   - 提交信息格式：
     Auto optimize website v001
   - 推送到当前分支。
   - 如果 git push 失败，在报告中说明失败原因，不要反复强推。

重要限制：
- 不要删除用户未确认的文件。
- 不要重置仓库。
- 不要提交运行前已经存在的未提交修改。
- 不要把 node_modules、dist、临时缓存或本机绝对路径资源提交进去。
- 不要提交 hatch-pet-runs、node_modules/.vite 等临时生成目录。
- 不要为了视觉效果牺牲明显性能。
```

## 截图命名规则

格式：

```text
screenshots/auto-YYYY-MM-DD-HHmm-vNNN-section.png
```

示例：

```text
screenshots/auto-2026-06-05-2330-v001-home.png
screenshots/auto-2026-06-05-2330-v001-about.png
screenshots/auto-2026-06-05-2330-v001-works.png
```

## 版本号规则

- 第一次自动化运行：`v001`
- 第二次自动化运行：`v002`
- 后续依次递增

如果自动化无法判断上一次版本号，检查：

- `reports/automation/`
- `screenshots/auto-*`
- 最近一次自动化提交信息

## 手动执行建议

当你想手动触发时，可以在 Codex 中说：

```text
按 Codex网站优化自动化.md 中的流程，手动执行一次网站自动优化。
```

## 安全建议

当前你选择了“当前项目目录 + 自动提交并推送”。这个效率高，但风险也更高。

建议遵守：

- 每次自动化运行前保持 git 工作区干净。
- 如果你正在手动改网页，先暂停自动化。
- 前几次运行后人工检查报告和截图。
- 稳定后再让自动化长期每天运行。
