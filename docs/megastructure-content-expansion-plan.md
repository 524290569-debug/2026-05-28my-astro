# 巨构主义内容扩展升级方案

版本基线：`v1.0.0`

目标版本建议：`v1.1.0`

## 1. 升级目标

当前巨构主义页面的基础功能已经成型：场景切换、路径返回、评论、审核、数据库记录都能支撑后续内容扩展。下一阶段不再优先做按钮、边框、装饰层这些细节，而是补足“世界感”。

这次升级的核心目标：

- 增加高质量巨构场景图，让页面从 3 个背景扩展成一个可探索的空间网络。
- 让场景之间有明确位置关系，用户点击节点像在进入一个迷宫，而不是切换图片轮播。
- 保留现有评论、审核、访问记录能力，并让新场景天然拥有独立评论空间。
- 控制 UI 信息密度，避免回到“满屏说明和装饰”的状态。

## 2. 当前状态判断

当前 `/megastructure/` 的内容结构集中在 `src/pages/megastructure.astro` 顶部：

- `sceneImagePaths`：3 张主背景图。
- `scenes`：3 个场景，分别是中庭、高原、光井。
- `nodes`：每个场景 1 个可点击节点，点击后进入下一个场景。
- `commentPageForScene(sceneId)`：评论按照 `megastructure:${scene.id}` 分开。
- `megastructure_visits.scene`：访问记录会记录当前场景 id。

这说明扩展场景时不需要重做评论或数据库结构。只要新场景有稳定的 `scene.id`，评论和访问记录就能自然隔离。

当前主要短板：

- 场景数量太少，路径关系偏线性。
- 远景、外部城市、地下空间、服务层等类型不够完整。
- 场景之间缺少“地理关系”，用户还没有在一个巨大空间里移动的感觉。
- 现有图片承担了太多表达，缺少一组统一风格的世界资产。

## 3. 场景扩展方向

建议第一轮新增到 6 个核心场景。保留现有 `atrium`、`highland`、`lightwell` 三个 id，避免旧评论失去归属；新增 3 个场景 id。

| 场景 id | 中文名 | 类型 | 作用 |
| --- | --- | --- | --- |
| `atrium` | 中庭 | 室内中庭 / 起点 | 保留作为入口和回路中心 |
| `highland` | 高原 | 高原巨构 / 外部远景 | 拉开尺度，承接用户参考图的空旷、神圣感 |
| `ring-city` | 环城 | 城市环形结构 | 强化城市尺度和工程秩序 |
| `cooling-vault` | 冷却层 | 地下冷却 / 服务空间 | 增加向下探索的压迫感和湿冷工业感 |
| `lightwell` | 光井 | 垂直光井 / 服务核心 | 保留垂直神圣光束，作为室内核心场景 |
| `far-towers` | 远塔群 | 远景塔群 / 终端眺望 | 形成宏大远方目标，补足世界边界 |

## 4. 图片资产方案

建议统一放在：

`public/assets/megastructure/`

建议文件命名：

- `scene-atrium-interior-v2.webp`
- `scene-highland-towers-v2.webp`
- `scene-ring-city-v1.webp`
- `scene-cooling-vault-v1.webp`
- `scene-lightwell-core-v2.webp`
- `scene-far-tower-cluster-v1.webp`

图片规格建议：

- 横屏主图优先，`1920x1080` 或接近 16:9。
- 每张压缩成 WebP，目标控制在 `120KB - 450KB`。
- 避免图片里出现文字、logo、水印、字幕。
- 风格统一为冷灰、蓝灰、湿雾、低饱和、巨大尺度、远处弱光。
- 不要每张都做成同一个蓝色调，允许高原和远塔带一点冷绿/晨光，冷却层带一点暗青，光井带白色强光。

## 5. 每张图的生成方向

### 5.1 中庭 `atrium`

定位：入口、内部空腔、混凝土、湿雾、回路中心。

画面要点：

- 巨大的室内中庭，不是普通走廊。
- 地面有积水反射，远处出口不明确。
- 混凝土柱体巨大但克制，不要太科幻飞船化。
- 适合承载评论点和入口节点。

### 5.2 高原 `highland`

定位：空旷、高原盆地、塔群从湿冷地表垂直升起。

画面要点：

- 远景尺度要大，参考用户给的巨构 ai / bilibili 图的感觉。
- 前景可以有山坡、湿草地、冷雾。
- 中远景是密集而高耸的塔群，像城市从地面被拉起来。
- 氛围偏神圣、空旷，后续可以适配 `Running Up That Hill（氛围版）`。

### 5.3 环城 `ring-city`

定位：环形城市结构、交通环、城市秩序。

画面要点：

- 俯视或高角度，能看出巨大的环形道路或城市骨架。
- 中央可以是空地、水面、冷却池或无人的绿地。
- 塔楼围绕环结构生长，形成明确的空间秩序。
- 不要做成普通赛博朋克夜景，重点是工程尺度。

### 5.4 冷却层 `cooling-vault`

定位：地下空间、冷却塔、管道、雾气、水汽。

画面要点：

- 地下或半地下巨型服务空间。
- 大尺度冷却结构、管道、格栅、潮湿地面。
- 视觉压低，空间更压迫。
- 适合作为从高原或环城向下进入的反差场景。

### 5.5 光井 `lightwell`

定位：垂直光井、室内核心、强光切开雾气。

画面要点：

- 一束或多束垂直白光从上方切入。
- 周围是高耸服务结构或内壁塔楼。
- 光要有体积感，但不能亮到看不清结构。
- 作为从地下冷却层向上回到核心的节点。

### 5.6 远塔群 `far-towers`

定位：远方目标、城市边界、终端眺望。

画面要点：

- 远处塔群像山一样竖立，前景有空旷地表。
- 塔群不需要太清晰，保持雾中神圣感。
- 适合作为探索终点或回望整座巨构的场景。
- 和高原不同：高原是接近巨构，远塔群是看到世界边界。

## 6. 路径关系设计

建议不要做成单线循环，而是轻量迷宫结构。每个场景保留 1 到 2 个节点，底部只显示当前场景和返回节点，避免信息过载。

建议关系：

```text
中庭 atrium
  -> 高原 highland
  -> 冷却层 cooling-vault

高原 highland
  -> 环城 ring-city
  -> 远塔群 far-towers

环城 ring-city
  -> 光井 lightwell
  -> 中庭 atrium

冷却层 cooling-vault
  -> 光井 lightwell
  -> 中庭 atrium

光井 lightwell
  -> 远塔群 far-towers
  -> 冷却层 cooling-vault

远塔群 far-towers
  -> 中庭 atrium
```

设计意图：

- `atrium` 是入口，不是唯一中心。
- `highland` 负责拉开尺度。
- `ring-city` 负责展示城市组织方式。
- `cooling-vault` 负责地下层次。
- `lightwell` 负责垂直核心。
- `far-towers` 负责远方目标和世界边界。

## 7. 代码改动范围

第一轮实现建议只改这些地方：

- `public/assets/megastructure/`
  - 新增 6 张主场景图。
- `src/pages/megastructure.astro`
  - 扩展 `sceneImagePaths`。
  - 扩展 `scenes`。
  - 扩展 `nodeImagePaths`。
  - 扩展 `nodes`。
  - 扩展 `nodeQuality` 和 `nodeViewport`。
  - 保持 `commentPageForScene(sceneId)` 不变。
- `src/styles/global.css`
  - 如有必要，仅微调节点在不同图片上的可见性。
  - 不新增大面积 UI 装饰。

暂时不改：

- Supabase 数据表结构。
- 评论审核逻辑。
- 登录注册逻辑。
- 音乐播放逻辑。
- 首页灰雨电台。

## 8. 文字内容规范

场景文案要短，不要解释太多。

建议结构：

- `label`：2 到 3 个中文字符，例如“中庭”“高原”“环城”。
- `kicker`：一行世界观标签，例如“高原盆地 / 远距塔群 / 无人环线”。
- `status`：英文技术感短语，例如 `HIGHLAND / OUTER CITY`。
- `description`：一句场景描述，控制在 24 到 42 个中文字符。
- `routeLabel`：描述将进入的目标，例如“进入地下冷却层”。
- `sentence`：目标场景的描述，而不是当前场景描述。

这样能保持页面轻，不会重新变成说明书。

## 9. 评论与访问记录兼容

评论隔离规则已经存在：

```js
commentPageForScene(sceneId) => `megastructure:${scene.id}`
```

所以新增场景后：

- `ring-city` 的评论只会存在 `megastructure:ring-city`。
- `cooling-vault` 的评论只会存在 `megastructure:cooling-vault`。
- `far-towers` 的评论只会存在 `megastructure:far-towers`。

保留旧场景 id 的原因：

- `atrium`、`highland`、`lightwell` 的旧评论不会丢。
- 访问记录里的旧场景名仍然可解释。

## 10. 性能策略

图片变多后要避免首屏变慢。

建议：

- 首屏只强预加载当前入口图。
- 其他场景图用浏览器自然加载，或者点击前再加载。
- 每张图片压缩为 WebP。
- 生成图尽量一次成型，少用后期叠加大量滤镜。
- 不引入重型图库或视频背景。

如果后续要做动态，可以先用 CSS 雾层和轻微视差，不要直接上大视频。

## 11. 验收标准

第一轮内容扩展完成后，至少要满足：

- 页面有 6 个可进入场景。
- 每个场景都有独立背景图，不再是 3 张图重复使用。
- 节点描述和目标场景一致。
- 返回节点仍能回到上一个场景。
- 评论按场景隔离。
- 管理员评论审核面板仍正常。
- `npm run build` 通过。
- 本地浏览器检查桌面和移动端不出现文字遮挡、节点明显错位、背景空白。

## 12. 推荐执行顺序

1. 先生成 6 张候选图，统一风格，不立刻改代码。
2. 从候选图里挑 6 张最稳的，压缩成 WebP 放入 `public/assets/megastructure/`。
3. 修改 `scenes` 和 `nodes`，接入 6 场景迷宫路径。
4. 检查评论隔离和访问记录是否自动沿用新场景 id。
5. 跑 `npm run build`。
6. 开本地服务检查 `/megastructure/`。
7. 如果效果成立，提交为 `v1.1.0` 方向的第一批内容更新。

## 13. 暂定结论

这次升级的重点不是“把当前页面修得更精致”，而是让它拥有更多地点。现在的功能底座已经够用，下一步应该优先补高质量场景资产和空间关系。

简单说：先把这个世界做大，再考虑它怎么动起来。否则动态再多，也只是在三张图上晃。[推眼镜]
